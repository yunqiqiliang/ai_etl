云器 Lakehouse 解决方案

多模态 AI ETL：大规模 LLM 批量推理

基于云器 Lakehouse 的结构化表和非结构化 Volume，通过 LLM Batch API 实现 50% 成本的大规模 AI 数据转换，结果自动写回 Lakehouse 并附带完整元数据。

业务挑战

企业在将 AI 能力融入数据处理流程时，面临以下核心问题：

💰

成本高昂

实时 API 按标准价格计费，处理数万条数据的推理费用难以控制。

⚡

并发受限

实时 API 有 QPM/TPM 限制，自行管理并发、限流和重试逻辑复杂且易出错。

🔗

数据断裂

推理结果散落在本地文件或临时存储中，无法与源数据关联，缺少审计元数据。

🖼️

多模态困难

图片、视频、音频等非结构化文件需要额外的存储管理和 URL 生成，流程碎片化。

解决方案概述

本方案将云器 Lakehouse 的湖仓一体能力与 LLM Batch API（DashScope / ZhipuAI）结合，构建端到端的多模态 AI ETL 流水线：

核心思路：从 Lakehouse 读取数据 → 构建 JSONL → 提交 Batch 推理（50% 成本）→ 结果 + 元数据写回 Lakehouse。双数据源并行提交，统一轮询等待。

为什么选择 Batch 推理？

LLM 服务商（如 DashScope、ZhipuAI）提供实时 API 和 Batch API 两种调用方式。对于大规模数据转换场景，Batch 模式在可靠性和完整性方面具有根本性优势：

🛡️

可靠性

服务端统一调度、限流和重试。实时 API 在第 8,000 条网络中断即丢失进度；Batch API 保证每条请求都被处理，失败的单独报告在 error 文件中。

✅

完整性

每条请求都有独立的 status_code 和完整审计信息（模型、token、finish_reason）。不会因客户端崩溃而静默丢失数据。

🔄

可恢复性

客户端进程中断后，Batch 任务在服务端继续运行。凭 batch_id 随时恢复，零数据丢失、零重复处理。

💰

成本优势

Batch API 统一定价为实时 API 的 50%。处理 5 万条数据，成本直接减半。

方案流程

Extract — 从 Lakehouse 读取数据

Table 模式：通过 ZettaPark SDK 执行 SELECT key, text FROM table WHERE filter 读取结构化文本。
Volume 模式：通过 DIRECTORY() 发现文件 → 扩展名过滤 → 增量过滤（跳过已处理）→ GET_PRESIGNED_URL() 生成 HTTP URL。

Transform — 构建 JSONL 并提交 Batch 推理

将源数据编码为 Batch API 要求的 JSONL 格式（每行一个请求，含 custom_id 用于结果回溯）。文本模式和多模态模式的 JSONL 结构不同，自动适配 Provider 的 content 格式。双数据源并行提交，两个 Batch 任务同时创建。

Wait — 统一轮询等待

所有 Batch 任务在一个轮询循环中并行等待。先完成的任务立即进入下载和写入阶段，不阻塞其他任务。支持中断后通过 resume 命令恢复。

Load — 结果 + 元数据写回 Lakehouse

推理结果解析后写入各自的目标表。每行包含推理结果和 12 个元数据字段（模型、token 消耗、处理时间、状态码等）。目标表不存在时自动创建，列缺失时自动添加。

快速开始

安装

git clone https://github.com/yunqiqiliang/ai_etl.git
cd ai_etl && pip install -e .

配置凭证（.env）

cp .env.example .env
# 编辑 .env，填入 API Key 和 Lakehouse 密码

配置 ETL 参数（config.yaml）

cp config.yaml.example config.yaml
# 编辑 config.yaml，配置 provider、源表/Volume、目标表、prompt

运行

python -m ai_etl run

Batch 推理通常需要 10–60 分钟。可随时 Ctrl+C，后续用 python -m ai_etl resume <batch_id> 恢复。
