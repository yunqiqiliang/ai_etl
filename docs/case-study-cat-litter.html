AI ETL Reference Case Study

猫砂竞品分析：从 Excel 到运营洞察

从 5 个电商平台的 Excel 销售数据出发，通过云器 Lakehouse + LLM Batch 推理，自动生成 862 条竞品结构化主数据，构建跨渠道汇总表，修复数据质量问题，最终输出可直接指导运营决策的竞品分析报告。

📋 业务背景

某宠物用品品牌（福丸）需要建立猫砂品类的竞品价盘数据库，覆盖抖音、天猫、天猫超市、京东自营、京东POP 五个电商平台，最终目标是输出可指导日常运营的竞品分析洞察。

核心挑战

各平台的商品名称是营销标题（如"福丸12包栀子花猫砂木薯猫砂混合豆腐猫砂去味除臭2.5kg"），需要从中提取结构化的产品主数据：品牌、类目、标准化产品名、配比、规格。人工处理 862 条数据耗时且不一致。更重要的是，各平台的销售数据格式完全不同（精确值 vs 区间值 vs 指数值），需要统一处理后才能做跨渠道对比分析。

导入方式

使用 Python openpyxl 读取 Excel + ZettaPark SDK 写入 Lakehouse，自动处理列名清洗、空行过滤、类型统一（全部 STRING）。

from ai_etl.lakehouse import LakehouseClient
lh = LakehouseClient(config=cfg)
lh.session.sql("CREATE SCHEMA IF NOT EXISTS cat_litter COMMENT '猫砂竞品分析'").collect()
# 每个 sheet → 一张表，自动建表 + 写入
df = lh.session.create_dataframe(data_rows, schema=schema)
df.write.save_as_table("cat_litter.sales_jd_self", mode="append")
💡所有表和字段都添加了中文注释（ALTER TABLE SET COMMENT / ALTER COLUMN COMMENT），方便在 Lakehouse Studio 中浏览。

发现的数据质量问题（5 个关键问题）

🚨数据正确是分析的基本前提。如果不修复这些问题，所有分析结论都是误导性的。
#问题原因影响修复方式
1 VOID 类型错误 CTAS 用抖音数据建表，抖音的浏览量/转化率为 NULL，ClickZetta 推断列类型为 VOID 后续 INSERT 天猫超市的 DOUBLE 值时报错：implicit cast not allowed for double to void 改用 CREATE TABLE 显式定义 schema（所有数值列为 DOUBLE）
2 京东件单价异常 京东的 成交客单价 字段有 ￥10万~￥20万 的值，解析为 150,000 件单价最高达 15 万元，严重拉高均价，导致价格带分析完全失真 件单价 > 5,000 元置为 NULL（猫砂单件不可能超过 5,000 元）
3 天猫超市转化率 > 100% 天猫超市的 转化指数 是相对指数值（如 3,500），不是百分比 如果放入转化率列，会出现 3,500% 的转化率 转化指数不放入转化率列，标记数据精度为"指数值(非百分比)"
4 天猫销量全部固定 37,500 天猫的 支付单量 是区间值（如"2.5万~5万"），之前用同一个中位数 37,500 填充所有行 50 个天猫商品的销量完全相同，无法区分畅销品和滞销品 每行独立解析各自的 支付单量 区间中位数

⑤ 运营建议

基于以上数据分析，为福丸品牌提出以下运营建议。

🔴 紧急：扩充京东自营 SKU

福丸在京东自营仅 5 个 SKU，远低于许翠花（14）和 pidan（16）。京东自营是高价值渠道（均价 ¥105，转化率 26.5%），扩充 SKU 是提升销售额最直接的手段。建议优先上架木薯猫砂（PB-TP）和混合猫砂（HB-TTCL）品类。

🟡 重要：优化天猫定价策略

天猫是福丸最大渠道（¥1,032 万），但均价仅 ¥68，低于京东自营的 ¥105。100–200 元价格带在京东的销售额最高（¥2,520 万），建议在天猫也推出更多中高端 SKU，提升客单价。

🔵 机会：抖音差异化选品

抖音渠道豆腐膨润土混合猫砂（HB-TFCL）销售额 ¥118 万，是抖音主力品类。福丸在抖音有 43 个 SKU 但单 SKU 产出低，建议聚焦 HB-TFCL 品类的爆款打造，减少长尾 SKU。

🟢 关注：高转化商品学习

京东 POP 中部分小品牌转化率超过 60%（如多恩 77.5%、洁客 67.5%），远高于行业均值。建议分析这些高转化商品的详情页、价格策略和评价，提炼可复用的转化优化经验。

💡 关键经验

1. Few-shot 示例决定 AI 质量

福丸的 46 条人工标注数据不是"要处理的数据"，而是"教 AI 怎么做的示例"。3 个精选的 few-shot 示例（覆盖不同品牌和类目）比 10 个同质示例效果更好。

2. 类目体系必须封闭

把 16 个类目编码作为"选择题"嵌入 prompt，而不是让 AI 自由发挥。这确保了 100% 的类目匹配率，且与 category_mapping 表完全对齐。

3. 数据正确是分析的前提

跨渠道汇总暴露了 5 个关键数据质量问题（VOID 类型、件单价异常、转化指数误用、销量固定值、转化率越界）。不修复就分析 = 误导决策。数据质量检查脚本应该在每次数据更新后自动运行。

4. 各平台数据格式差异巨大

抖音是精确值，天猫是区间值，天猫超市是指数值，京东是区间值。统一处理时必须标记"数据精度"列，避免将指数值当作真实金额做跨渠道对比。

5. 显式 schema 避免类型推断陷阱

ClickZetta 的 CTAS 会将 NULL 列推断为 VOID 类型，后续 INSERT 非 NULL 值时报错。解决方案：先 CREATE TABLE 显式定义所有列类型，再 INSERT 数据。

6. Batch API 保证大规模完整性

862 条请求 100% 成功，0 失败。Batch API 的服务端重试机制在大规模处理时比实时 API 更可靠。50% 成本折扣在数据量大时节省显著。
